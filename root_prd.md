# root_prd.md

## 1. 文档定位

本文件是 `build-from-scratch` 仓库的根级产品与实现规范文档，用于约束本仓库下所有“从 0 开始实现”的项目。

本文件描述的是全局规则，不针对某一个具体算法、模型或工程模块。

除非用户明确要求，否则后续不要修改本文件。

---

## 2. 项目目标

`build-from-scratch` 的目标是：

1. 用尽量少的外部依赖，完成常见算法、模型、工具或流程的从零实现。
2. 让读者不仅能“调用”，还能够看到核心过程、关键数据结构和运行步骤。
3. 保证代码结构清晰、过程可运行、注释完整、适合学习和演示。
4. 区分“演示调用”和“底层从零实现”，避免把外部库封装一下就当作 scratch 实现。
5. 以教学为优先目标，帮助初学者真正理解实现过程，而不是展示写代码的人有多熟练。

---

## 3. 全局目录约定

对于仓库中的任意具体主题，都应尽量遵循以下模式：

1. 根目录的 `root_prd.md` 是全局总规范。
2. 某个具体主题应有自己的独立目录。
3. 某个具体主题如果需要单独 PRD，则文件名使用 `<topic>_prd.md`。
4. 例如：
   - `bpe/bpe_prd.md`
   - `svm/svm_prd.md`
   - `rf/rf_prd.md`

命名规则说明：

1. 根目录总 PRD 固定命名为 `root_prd.md`。
2. 子项目 PRD 固定命名为 `<项目名>_prd.md`。
3. 后续新增具体项目时，应优先新增对应子项目 PRD，而不是修改 `root_prd.md`。
4. 除非用户明确要求，否则不要修改 `root_prd.md`。

---

## 4. 实现分层原则

每一个具体项目，原则上分成两层：

### 4.1 最外层演示层

最外层可以写直接可运行的示例，用来给读者展示“怎么使用”。

这一层可以适度调用 Python 标准能力，或者在特定情况下调用成熟库做对照演示，但必须满足以下约束：

1. 这一层的作用是“展示使用方式”，不是“替代 scratch 实现”。
2. 如果使用了外部成熟库，必须明确标注这是演示层，不属于底层从零实现。
3. 这一层不能掩盖真实实现过程。

例如：

1. 可以写一个最外层示例，展示 `svm` 或 `rf` 的常见调用方式。
2. 但不能把这种调用当成 `build_wheel` 中的 scratch 实现。
3. 更不能直接通过现成库完成核心逻辑后，伪装成从零实现。

### 4.2 Scratch 实现层

真正的从零实现，应放在对应项目目录下的 `build_wheel/` 中。

要求：

1. `build_wheel/` 是该项目的 scratch 实现主目录。
2. `build_wheel/` 中必须有 `main.py`。
3. 运行 `main.py` 后，应能看到该项目核心流程的完整演示。
4. 演示应尽量覆盖关键中间步骤，而不只是输出最终结果。
5. 代码应体现真实实现过程，而不是对外部现成实现做简单包装。

---

## 5. 依赖约束

本仓库强调“从零实现”，因此依赖策略如下：

### 5.1 允许使用的内容

允许使用以下内容作为基础工具：

1. Python 标准库。
2. 与工程组织、字符串处理、文件处理、参数解析相关的基础能力。
3. 类似 `re` 这样的标准库模块。
4. 可以把某些足够基础、足够通用、不会掩盖核心算法思想的能力，当作不复现的最小单元。

### 5.2 不允许使用的内容

以下做法不符合本仓库目标：

1. 直接导入已经实现目标能力的现成库来完成核心逻辑。
2. 例如做 BPE 时，不能出现本质等同于 `import bpe` 的做法。
3. 不能把第三方库结果包一层打印和注释，就当作 scratch 实现。
4. 不能把最关键的算法步骤外包给现成工具。

### 5.3 关于“最小不复现单元”的原则

为了保持实现简洁，可以明确一些不需要重复制造轮子的基础单元，但必须满足：

1. 该单元不会遮蔽项目核心思想。
2. 该单元不会替代关键算法逻辑。
3. 使用它能显著降低样板代码，提高可读性。
4. 使用后，读者仍然能完整理解本项目真正要表达的核心过程。

后续每一个子项目 PRD 都必须明确写清楚：

1. 本项目的“最小粒度”或“最小不复现单元”是什么。
2. 哪些能力可以直接使用。
3. 哪些能力必须自己实现。
4. 这样划分的教学理由是什么。

---

## 6. 教学优先原则

本仓库首先是教学型仓库，不是炫技型仓库。

所有具体项目都必须把“让小白看懂”作为第一目标。

要求如下：

1. 代码结构要服务于理解，不要为了显得高级而故意写得抽象。
2. 变量命名、函数拆分、输出内容、注释方式都应优先考虑初学者的阅读体验。
3. 不要为了追求代码短、技巧多、写法巧而牺牲可读性。
4. 不要把多个步骤压成一行或一小段难以理解的逻辑。
5. 如果一种写法更短，但另一种写法更容易讲明白，应优先选择更容易讲明白的写法。
6. 输出结果要帮助读者理解过程，而不是只展示答案。

默认禁止的倾向包括：

1. 炫技式写法。
2. 过度抽象。
3. 为了优雅而隐藏关键步骤。
4. 对初学者不友好的命名和流程组织。

---

## 7. 架构设计要求

所有具体项目在开始编码前，必须先想清楚整体架构，禁止边写边补、反复打补丁式开发。

要求如下：

1. 先明确输入、输出、核心流程、关键数据结构。
2. 先拆分模块职责，再开始实现。
3. `main.py` 负责串联流程与展示过程，不应承担全部底层细节。
4. 核心逻辑、辅助函数、数据处理、可视化输出应尽量分层组织。
5. 设计时优先考虑“教学清晰度”和“运行过程可观察性”。

不接受的方式：

1. 先堆一版能跑的代码，后续再不断修补结构。
2. 把所有逻辑都塞进一个超长脚本。
3. 命名混乱、职责不清、流程不可读。

---

## 8. 运行与展示要求

每个 scratch 项目都应尽量做到“直接运行即可理解过程”。

要求如下：

1. `build_wheel/main.py` 必须可直接运行。
2. 运行后应展示：
   - 输入是什么
   - 核心步骤如何推进
   - 中间状态如何变化
   - 最终结果是什么
3. 输出内容要服务于教学，不要只给一个黑盒结果。
4. 如果流程较长，应分阶段打印，帮助读者理解。
5. 如果某些步骤为了简洁做了抽象，应在注释中明确说明。

---

## 9. 注释规范

本仓库要求中文注释完整，并分成两类：

### 9.1 开头注释

用于文件、模块、较大逻辑块的开头说明。

用途：

1. 说明这一段代码要解决什么问题。
2. 说明这一部分在整体流程中的位置。
3. 说明为什么这样设计。
4. 说明输入输出或关键注意点。

### 9.2 函数 / 长难句注释

用于函数定义前，或复杂语句、关键步骤附近。

用途：

1. 解释函数职责。
2. 解释复杂逻辑为什么这么写。
3. 解释不直观的数据结构变化。
4. 解释长逻辑链条中的关键转折。

注释要求：

1. 注释必须使用中文。
2. 注释要解释“为什么”和“在做什么”，不要只重复代码字面意思。
3. 简单直白的一行赋值不必强行注释。
4. 长函数、复杂循环、关键判断、核心公式附近必须有解释。

---

## 10. 子项目 PRD 的必写内容

后续每一个具体项目的 PRD，都必须至少写清楚以下内容：

1. 项目要解决什么问题。
2. 项目的教学目标是什么。
3. 面向的读者是谁，默认应包含初学者视角。
4. 本项目的最小粒度或最小不复现单元是什么。
5. 哪些库可以用，为什么可以用。
6. 哪些库不可以用，为什么不可以用。
7. 哪些核心步骤必须自己实现。
8. 项目的目录结构和模块职责。
9. `build_wheel/main.py` 将如何展示完整过程。
10. 注释和输出应如何帮助读者理解。

例如：

1. 做 BPE 时，在 `bpe/` 目录下新增 `bpe_prd.md`。
2. `bpe_prd.md` 必须明确说明：
   - BPE 的教学目标是什么
   - token 的最小粒度是按什么定义的
   - 哪些文本清洗步骤可以直接用标准库
   - 哪些合并逻辑必须自己实现
   - 哪些依赖不能使用
3. `bpe_prd.md` 负责描述 BPE 项目的目标、范围、输入输出、模块拆分、演示流程与实现约束。

---

## 11. 针对具体项目的扩展规则

当仓库新增某个具体主题时，应新增该主题自己的 PRD 文件。

例如：

1. 做 BPE 时，在 `bpe/` 目录下新增 `bpe_prd.md`。
2. `bpe_prd.md` 是 BPE 项目的直接规范。
3. `root_prd.md` 仍然是全局规范，不应被子项目规则随意覆盖。

规则优先级：

1. 用户明确要求优先级最高。
2. 子项目 PRD 只约束对应项目。
3. 根目录 `root_prd.md` 约束整个仓库。
4. 如果子项目 PRD 与根级规则冲突，应先由用户决定是否允许例外。

---

## 12. 交付标准

一个合格的 scratch 项目，至少应满足：

1. 有清晰的项目级 PRD。
2. 有明确的目录结构。
3. `build_wheel/` 中有真实 scratch 实现。
4. `build_wheel/main.py` 可直接运行并展示关键过程。
5. 不依赖现成核心库完成主要逻辑。
6. 中文注释完整。
7. 架构先行，而不是后期修补。
8. 读者能通过代码和输出看到“它是怎么一步一步做出来的”。
9. 初学者能够顺着代码和输出理解核心思路，而不是只能看结果。

---

## 13. 默认执行规则

除非用户明确提出例外，否则后续新增或实现任何具体项目时，默认遵守本文件。

尤其是：

1. 根目录总 PRD 文件名固定为 `root_prd.md`。
2. 子项目 PRD 文件名固定为 `<项目名>_prd.md`。
3. 除非用户明确要求，否则不要修改 `root_prd.md`。
4. 真正的 scratch 实现放在对应项目的 `build_wheel/` 中。
5. `build_wheel/main.py` 必须可运行并展示完整过程。
6. 中文注释必须完整，并区分开头注释与函数/长难句注释。
7. 开发前先定架构，不接受修修补补式实现。
8. 子项目 PRD 必须写清楚教学目标、最小粒度、允许依赖、禁止依赖。
9. 所有实现默认以“教明白小白”为优先目标，而不是炫技。
