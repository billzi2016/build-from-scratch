# bpe_prd.md

## 1. 文档定位

本文件是 `bpe/` 项目的专属 PRD。

本项目受根目录 [root_prd.md](/Users/bizi/Desktop/GitHub/build-from-scratch/root_prd.md) 约束，但在实现边界上与 `imagenet-ai` 不同。

`bpe/` 是算法型项目，重点是把 `BPE` 的核心流程从头讲明白，因此这里不能走“只展示架构、底层交给框架”的路线。

---

## 2. 项目目标

`bpe/` 的目标是：

1. 用尽量少的依赖，从头实现一个教学版 BPE。
2. 让初学者看懂 BPE 为什么要做子词切分。
3. 让初学者看懂 BPE 如何统计相邻单元频率、如何逐步合并、如何形成词表。
4. 让读者能从输入文本、清洗结果、训练过程、合并步骤、最终编码结果一路看明白。
5. 代码重点是教学，不是追求极限性能。

---

## 3. 教学定位

本项目首先是给初学者讲明白 BPE 的项目。

要求如下：

1. 先讲清概念，再讲实现。
2. 先讲清每一步为什么做，再讲代码怎么写。
3. 输出必须帮助读者看到中间过程。
4. 不要为了写法高级而把流程藏起来。
5. 不要为了性能把教学性牺牲掉。

---

## 4. BPE 项目的 from scratch 边界

`bpe/` 和 `imagenet-ai` 的边界不同。

这里的 `from scratch` 指的是：

1. BPE 的核心算法必须自己实现。
2. 词表构建逻辑必须自己实现。
3. 合并规则学习过程必须自己实现。
4. 编码和解码逻辑必须自己实现。

不允许直接调用已经实现好 BPE 的库来替代核心过程。

换句话说：

1. 这里不能 `import bpe`
2. 不能直接调用现成 tokenizer 库的 BPE 训练器
3. 不能把第三方结果包一层打印就当成 scratch 实现

---

## 5. 最小粒度 / 最小不复现单元

本项目允许有“最小不复现单元”，但这个边界要非常保守。

### 5.1 允许直接使用的最小单元

允许直接使用：

1. Python 标准库
2. `re`
3. `json`
4. `collections`
5. `itertools`
6. `pathlib`
7. `argparse`
8. `math`

这些能力可以作为基础工程工具使用，因为它们不会替代 BPE 的核心思想。

### 5.2 必须自己实现的部分

以下内容必须自己写：

1. 初始 token 单元的定义方式
2. 训练语料的切分表示
3. 相邻 token 对频率统计
4. 最优合并对的选择
5. merge 规则更新
6. 词表增长过程
7. 文本编码逻辑
8. 文本解码逻辑

### 5.3 本项目的最小粒度说明

本项目默认把“字符串处理”和“基础正则清洗”视为不复现的最小单元。

但 BPE 真正的最小教学粒度应放在：

1. 字符级初始表示
2. 相邻 pair 统计
3. merge 规则学习
4. 子词编码结果

也就是说：

1. 可以用 `re` 做基础清洗
2. 不能用现成 tokenizer 代替 pair merge 过程

---

## 6. 可以使用的库

默认允许：

1. Python 标准库
2. `re`
3. `json`
4. `collections`
5. `itertools`
6. `pathlib`
7. `argparse`
8. `typing`

默认不需要引入任何第三方依赖。

如果后续用户明确允许，可以为可视化或测试增加少量辅助依赖，但默认版本不依赖第三方 tokenizer 生态。

---

## 7. 不可以使用的库或方式

默认不允许：

1. `tokenizers`
2. `sentencepiece`
3. `tiktoken`
4. 任何现成 BPE 训练器
5. 任何现成 tokenizer 模型文件直接加载后拿来演示

也不允许：

1. 直接调用第三方库得到 merges
2. 直接调用第三方库得到 vocab
3. 把最关键的 merge 学习过程外包

---

## 8. 项目范围

本项目默认做“教学版 BPE”，不追求生产级 tokenizer 的所有边界问题。

默认覆盖内容：

1. 文本基础清洗
2. 初始字符级切分
3. 词频统计
4. pair 频率统计
5. merge 学习
6. vocab 构建
7. encode
8. decode
9. 训练过程可视化打印

默认不优先处理的复杂主题：

1. Unicode 全边界细节
2. 超大规模语料优化
3. 并行训练
4. 工业级压缩优化
5. 各类 tokenizer 兼容格式导出

## 9. 推荐目录结构

推荐结构如下：

```text
bpe/
├── bpe_prd.md
└── build_wheel/
    ├── main.py
    ├── cleaner.py
    ├── trainer.py
    ├── encoder.py
    ├── decoder.py
    ├── vocab.py
    └── utils.py
```

如果后续实现规模更小，也可以适度合并文件，但必须保证教学结构清晰。

---

## 10. `build_wheel/main.py` 的职责

`main.py` 必须做到“运行一次就能看到整个过程”。

至少应展示：

1. 原始输入文本
2. 清洗后的文本
3. 初始 token 形式
4. 每轮 merge 的最佳 pair
5. vocab 如何增长
6. 最终 merges
7. 示例文本的 encode 结果
8. 示例 token 序列的 decode 结果

`main.py` 不应该只是打印最终词表。

---

## 11. 模块职责建议

建议职责如下：

1. `cleaner.py`
   负责文本基础清洗和标准化。

2. `trainer.py`
   负责 BPE 训练主流程，包括 pair 统计和 merge 学习。

3. `vocab.py`
   负责词表相关操作。

4. `encoder.py`
   负责把文本按学到的 merge 规则编码成 token 序列。

5. `decoder.py`
   负责从 token 序列还原文本。

6. `utils.py`
   负责通用辅助函数。

---

## 12. 注释规范

本项目必须使用中文注释，并遵守根目录注释规范。

至少包括：

1. 文件开头注释
2. 函数前注释
3. 长难句或关键逻辑附近注释

重点必须解释：

1. 当前这一步在 BPE 流程里的位置
2. 为什么这样定义最小 token 单元
3. 为什么这个 pair 会被选中
4. merge 之后数据结构发生了什么变化

---

## 13. 交付标准

一个合格的 `bpe/` 实现，至少应满足：

1. 不依赖现成 BPE 库
2. 能从头训练出教学版 merge 规则
3. 能输出清晰的中间训练过程
4. 能进行 encode
5. 能进行 decode
6. `main.py` 可直接运行
7. 中文注释完整
8. 初学者能顺着输出理解 BPE 每一步在做什么

---

## 14. 默认执行规则

除非用户明确要求例外，否则后续在 `bpe/` 中开发时默认遵守：

1. 优先教学清晰度，不优先性能优化
2. 优先完整展示 merge 过程，不优先写成黑盒
3. 允许使用标准库做基础工程工作
4. 不允许直接调用现成 BPE 能力替代核心算法
5. 真正的 scratch 边界放在 BPE 算法本身
